# EEVDF: Heap Corruption via min_deadline Propagation on Current Entity

**Commit:** `d2929762cc3f85528b0ca12f6f63c2a714f24778`
**Affected files:** kernel/sched/fair.c
**Fixed in:** v6.6-rc7
**Buggy since:** v6.6-rc6 (commit 8dafa9d0eb1a "sched/eevdf: Fix min_deadline heap integrity")

## Bug Description

In the EEVDF scheduler, the `min_deadline` field on each scheduling entity (`sched_entity`) maintains a min-heap augmentation of the red-black tree used for task ordering. Each node's `min_deadline` equals the minimum of its own `deadline` and the `min_deadline` values of its left and right children. This heap structure enables O(log n) lookup of the earliest-deadline eligible task during `pick_eevdf()`.

When the weight of a scheduling entity changes (e.g., because a task group's shares are recomputed), the function `reweight_entity()` adjusts the entity's deadline proportionally and must propagate this change up the augmented rb-tree to maintain the min-heap invariant. Commit 8dafa9d0eb1a correctly identified that this propagation was missing and added a call to `min_deadline_cb_propagate(&se->run_node, NULL)` after updating the deadline. However, it failed to account for a critical subtlety: the currently running entity (`cfs_rq->curr`) has `se->on_rq == 1` but is **not actually present in the rb-tree**. The `set_next_entity()` function explicitly removes `curr` from the tree via `__dequeue_entity()` with the comment: `/* 'current' is not kept within the tree. */`.

When `min_deadline_cb_propagate()` is called on the `run_node` of the current entity, it attempts to walk up the tree via `rb_parent(&node->run_node)`. Since the node was previously removed from the tree by `rb_erase_augmented_cached()`, its `rb_parent` pointer is stale—it still points to whatever node was its parent at the time of removal. If that parent's task has since been freed (e.g., the task exited and its `task_struct` was reclaimed), the stale pointer leads to freed slab memory, causing a KASAN slab-use-after-free. Even without KASAN, this corrupts the min_deadline heap by writing computed values to the wrong locations in the tree, breaking the O(log n) earliest-deadline lookup.

The bug was reported independently by syzbot (via syzkaller fuzzing) and by Dmitry Safonov, who managed to reproduce it locally on v6.6-rc6. The syzbot report shows the crash occurring in `reweight_entity+0x8e3/0xa60` at `__update_min_deadline` inlined into `min_deadline_cb_propagate`, triggered during the `entity_tick` → `update_cfs_group` → `reweight_entity` code path while a timer tick is processing the currently running task.

## Root Cause

The root cause is the unconditional call to `min_deadline_cb_propagate(&se->run_node, NULL)` in `reweight_entity()` without checking whether the entity is the current task of its CFS runqueue.

In `reweight_entity()`, the code distinguishes between on-runqueue and off-runqueue entities:

```c
if (!se->on_rq) {
    /* Scale vlag for off-rq entities */
    se->vlag = div_s64(se->vlag * old_weight, weight);
} else {
    /* Scale deadline for on-rq entities */
    s64 deadline = se->deadline - se->vruntime;
    deadline = div_s64(deadline * old_weight, weight);
    se->deadline = se->vruntime + deadline;
    min_deadline_cb_propagate(&se->run_node, NULL);  // <-- BUG
}
```

The `else` branch runs whenever `se->on_rq` is true. However, the currently running entity has `on_rq == 1` but is **not in the rb-tree**. This is a well-known design decision in the CFS/EEVDF scheduler: when a task is picked for execution in `set_next_entity()`, it is dequeued from the rb-tree:

```c
static void set_next_entity(struct cfs_rq *cfs_rq, struct sched_entity *se)
{
    /* 'current' is not kept within the tree. */
    if (se->on_rq) {
        __dequeue_entity(cfs_rq, se);  // removes from rb-tree
        ...
    }
    cfs_rq->curr = se;
}
```

After `__dequeue_entity()`, the `se->run_node` is detached from the tree. The rb-tree erase operation (`rb_erase_augmented_cached`) leaves the removed node's internal pointers (particularly `__rb_parent_color` which encodes the parent pointer) in an undefined state—they may still point to what was previously the parent node in the tree, or to the successor node if a swap occurred during erase.

The `min_deadline_cb_propagate()` function, generated by the `RB_DECLARE_CALLBACKS` macro, implements:

```c
static inline void
min_deadline_cb_propagate(struct rb_node *rb, struct rb_node *stop)
{
    while (rb != stop) {
        struct sched_entity *node = rb_entry(rb, struct sched_entity, run_node);
        if (min_deadline_update(node, true))
            break;
        rb = rb_parent(&node->run_node);
    }
}
```

When called on the detached `curr`'s `run_node`:
1. First iteration: `min_deadline_update()` reads `node->run_node.rb_left` and `node->run_node.rb_right`, which are stale pointers from the removed node. It computes `min_deadline` from these potentially invalid children and writes it to the detached node.
2. It then calls `rb_parent(&node->run_node)`, which extracts the parent pointer from the stale `__rb_parent_color` field. This yields a pointer to memory that may have been freed (if the parent entity's task exited) or that belongs to a completely unrelated slab allocation.
3. Subsequent iterations dereference these garbage pointers, reading and writing `min_deadline` values into arbitrary memory locations.

The specific call chain that triggers this is: `task_tick_fair()` → `entity_tick()` → `update_cfs_group(curr)` → `reweight_entity()`. The `task_tick_fair()` function iterates up the scheduling hierarchy via `for_each_sched_entity(se)`, and at each level, `se` is `cfs_rq->curr`. When `update_cfs_group()` detects that the group's shares have changed (via `calc_group_shares()`), it calls `reweight_entity()` on this `curr` entity, triggering the bug.

## Consequence

The most severe consequence is a **KASAN slab-use-after-free** crash. The syzbot report shows:

```
BUG: KASAN: slab-use-after-free in __update_min_deadline kernel/sched/fair.c:805 [inline]
BUG: KASAN: slab-use-after-free in min_deadline_update kernel/sched/fair.c:819 [inline]
BUG: KASAN: slab-use-after-free in min_deadline_cb_propagate kernel/sched/fair.c:825 [inline]
BUG: KASAN: slab-use-after-free in reweight_entity+0x8e3/0xa60 kernel/sched/fair.c:3660
Read of size 8 at addr ffff888022a59a70 by task syz-executor206/5331
```

The address `ffff888022a59a70` is 176 bytes inside a freed 8960-byte `task_struct` slab object. The freed task's `sched_entity.run_node` was the previous rb-tree parent of the current entity before it was dequeued. After the task exited and its `task_struct` was freed via RCU (`delayed_put_task_struct`), the stale parent pointer now points into freed slab memory.

Without KASAN enabled, the consequences are more insidious:

1. **Silent min_deadline heap corruption**: The `min_deadline_update()` function writes computed values through stale pointers. If those stale pointers happen to land in valid memory (e.g., a reallocated slab object), the writes silently corrupt the `min_deadline` fields of whatever data happens to occupy that memory. If the pointer happens to land within another active `sched_entity`, the min_deadline heap invariant is broken, leading to incorrect task selection in `pick_eevdf()`.

2. **Incorrect scheduling decisions**: A corrupted min_deadline heap causes `pick_eevdf()` to miss the task with the earliest deadline. This manifests as the "EEVDF scheduling fail, picking leftmost" warning (from the predecessor commit's reports), where the scheduler falls back to picking the leftmost rb-tree node because the heap-based search fails to find an eligible entity with the earliest deadline.

3. **Potential kernel panic**: If the stale pointer dereference hits an unmapped page, the kernel will oops with a page fault in interrupt context (since `entity_tick` runs from the timer interrupt), which typically leads to a kernel panic.

The bug is triggered on every timer tick where a cgroup task is running and its group's shares differ from the cached weight. In systems with active cgroup hierarchies (e.g., container orchestration with Docker/Kubernetes), this is a common occurrence, making the bug readily triggerable in production.

## Fix Summary

The fix is a one-line change that adds a guard condition before calling `min_deadline_cb_propagate()`:

```c
if (se != cfs_rq->curr)
    min_deadline_cb_propagate(&se->run_node, NULL);
```

This ensures that `min_deadline_cb_propagate()` is only called when the entity is actually present in the rb-tree. When `se == cfs_rq->curr`, the entity is not in the tree (it was removed by `set_next_entity()` → `__dequeue_entity()`), so propagation would walk stale pointers and must be skipped.

This fix is correct because when `se == cfs_rq->curr`, the entity's `min_deadline` does not participate in the augmented rb-tree's heap structure—it is not a node in the tree. The entity's deadline is still updated (the `se->deadline = se->vruntime + deadline` assignment remains), but propagation up the tree is skipped because there is no tree path to propagate along. When the entity is eventually re-enqueued into the tree (via `put_prev_entity()` → `__enqueue_entity()`), `rb_add_augmented_cached()` will correctly initialize `min_deadline` and propagate it up the tree.

The existing code already handles the `curr` case correctly in other places within `reweight_entity()`: the `if (cfs_rq->curr == se)` check at the top calls `update_curr()` instead of `avg_vruntime_sub()`, and the `if (cfs_rq->curr != se)` check at the bottom calls `avg_vruntime_add()` only for non-current entities. The missing `se != cfs_rq->curr` guard on the `min_deadline_cb_propagate()` call was simply an oversight in the predecessor fix (8dafa9d0eb1a).

## Triggering Conditions

The bug requires the following conditions to be met simultaneously:

1. **CONFIG_FAIR_GROUP_SCHED must be enabled**: This is required for `update_cfs_group()` to have any effect. Without group scheduling, `update_cfs_group()` is an empty inline function that returns immediately. This config option is enabled by default on most distribution kernels.

2. **A task group (cgroup) must exist with at least one running task**: The bug is triggered via the `entity_tick()` → `update_cfs_group()` path. The `update_cfs_group(se)` function only does work when `group_cfs_rq(se)` is non-NULL, which is true for group scheduling entities (not leaf task entities). This means the call chain must reach a group-level `sched_entity` in the hierarchy walk.

3. **The group's computed shares must differ from its current weight**: `update_cfs_group()` calls `calc_group_shares()` to compute the current shares and compares with `se->load.weight`. If they match, it returns without calling `reweight_entity()`. The shares change when the load on the group's CFS runqueue changes—for example, when tasks within the group wake up, go to sleep, or change their load characteristics. On SMP systems, `calc_group_shares()` depends on `gcfs_rq->avg.load_avg` (PELT load average), which changes gradually.

4. **The group sched_entity must be the current entity on its parent CFS runqueue**: This is naturally the case during `entity_tick()`, which processes the currently running entity at each level of the scheduling hierarchy. When `task_tick_fair()` calls `entity_tick()` for a group sched_entity, that entity is `cfs_rq->curr` on its parent runqueue.

5. **The group sched_entity's `run_node` must have a stale `rb_parent` pointer**: After `set_next_entity()` removes the entity from the rb-tree, the `rb_parent` pointer may point to a node whose task has since been freed. This happens naturally when another entity in the rb-tree (the former parent) is dequeued and its task exits. The more churn in the cgroup (tasks being created, destroyed, migrated), the more likely a stale pointer will reference freed memory.

6. **Kernel version between v6.6-rc6 and v6.6-rc7 (exclusive)**: The bug was introduced in v6.6-rc6 by commit 8dafa9d0eb1a and fixed in v6.6-rc7 by this commit.

The bug is reliably triggered by creating a cgroup with multiple tasks, having one task run while other tasks' state changes (wake/sleep), and letting the timer tick process the running task. In syzkaller's reproduction, it was triggered by having multiple processes performing file operations under cgroup constraints, causing natural task churn and group weight recalculations.

## Reproduce Strategy (kSTEP)

The goal is to trigger the `entity_tick()` → `update_cfs_group()` → `reweight_entity()` path for a group scheduling entity that is `cfs_rq->curr` and whose group shares have changed since the last tick.

### Step 1: QEMU Configuration

Configure QEMU with at least 2 CPUs (`-smp 2`). CPU 0 is reserved for the driver; tasks will be pinned to CPU 1.

### Step 2: Cgroup Setup

Create a cgroup and assign tasks to it:

```c
kstep_cgroup_create("testgrp");
kstep_cgroup_set_weight("testgrp", 100);
```

### Step 3: Task Creation

Create 3 CFS tasks and pin them all to CPU 1:

```c
struct task_struct *t1 = kstep_task_create();
struct task_struct *t2 = kstep_task_create();
struct task_struct *t3 = kstep_task_create();
kstep_task_pin(t1, 1, 1);
kstep_task_pin(t2, 1, 1);
kstep_task_pin(t3, 1, 1);
kstep_cgroup_add_task("testgrp", t1->pid);
kstep_cgroup_add_task("testgrp", t2->pid);
kstep_cgroup_add_task("testgrp", t3->pid);
```

### Step 4: Trigger Group Share Changes

The idea is to change the group's load between ticks so that `calc_group_shares()` returns a different value on the next `update_cfs_group()` call:

1. Wake all three tasks so they are runnable: `kstep_task_wakeup(t1)`, `kstep_task_wakeup(t2)`, `kstep_task_wakeup(t3)`.
2. Let several ticks pass so PELT load averages stabilize: `kstep_tick_repeat(100)`.
3. Block two tasks to change the group load: `kstep_task_block(t2)`, `kstep_task_block(t3)`.
4. Immediately tick to trigger `entity_tick()` → `update_cfs_group()` on the running group entity: `kstep_tick()`.

When the load changes suddenly (from 3 running tasks to 1), `calc_group_shares()` produces a significantly different value, causing `reweight_entity()` to be called on the group `sched_entity` while it is `cfs_rq->curr`.

### Step 5: Detection via Heap Validation

After the tick, validate the min_deadline heap property of the rb-tree on CPU 1's CFS runqueue. Use `KSYM_IMPORT` to access internal scheduler structures:

```c
KSYM_IMPORT(void, validate_cfs_rq, struct cfs_rq *cfs_rq);
```

Alternatively, manually walk the rb-tree and verify the augmented heap invariant:

```c
static bool check_min_deadline_heap(struct cfs_rq *cfs_rq)
{
    struct rb_node *node;
    for (node = rb_first_cached(&cfs_rq->tasks_timeline); node; node = rb_next(node)) {
        struct sched_entity *se = rb_entry(node, struct sched_entity, run_node);
        if (se->min_deadline > se->deadline)
            return false;
        if (node->rb_left) {
            struct sched_entity *left = rb_entry(node->rb_left, struct sched_entity, run_node);
            if (se->min_deadline > left->min_deadline)
                return false;
        }
        if (node->rb_right) {
            struct sched_entity *right = rb_entry(node->rb_right, struct sched_entity, run_node);
            if (se->min_deadline > right->min_deadline)
                return false;
        }
    }
    return true;
}
```

### Step 6: Detection via Kernel Crash

On the buggy kernel, if the stale `rb_parent` pointer dereferences unmapped memory, the kernel will panic during the tick. The driver can detect this by checking if ticks succeed without a crash. Use `kstep_pass()` if the tick completes and the heap is valid, or `kstep_fail()` if corruption is detected.

### Step 7: Pass/Fail Criteria

- **Buggy kernel (v6.6-rc6 with 8dafa9d0eb1a but without d2929762cc3f)**: The tick that triggers `reweight_entity()` on `cfs_rq->curr` either crashes the kernel (oops/panic from stale pointer dereference) or silently corrupts the min_deadline heap. If the kernel survives, the heap validation check will detect corruption. Report `kstep_fail()`.
- **Fixed kernel (v6.6-rc7 or later)**: The `if (se != cfs_rq->curr)` guard in `reweight_entity()` prevents the stale pointer walk. The tick completes normally and the heap remains valid. Report `kstep_pass()`.

### Step 8: Enhancing Reliability

To maximize the chance of detecting corruption (as opposed to the stale pointer accidentally landing in valid memory):

1. Repeat the wake/block/tick cycle multiple times (e.g., 50 iterations) to increase the chance that the stale `rb_parent` pointer causes observable damage.
2. Create more tasks in the cgroup and have them enter/exit frequently to increase the churn and likelihood of the stale pointer referencing freed memory.
3. After each tick cycle, run the heap validation check.
4. Additionally, the root cfs_rq's rb-tree (which holds the group scheduling entities) should also be validated, as the bug corrupts the parent-level tree, not the group-internal tree.

### Step 9: Callback Usage

Use the `on_tick_end` callback to perform the heap validation after each tick:

```c
void on_tick_end(void) {
    struct rq *rq = cpu_rq(1);
    struct cfs_rq *cfs_rq = &rq->cfs;
    if (!check_min_deadline_heap(cfs_rq)) {
        kstep_fail("min_deadline heap corrupted after reweight on curr");
    }
}
```

### Step 10: Additional Logging

Add `printk()` logging inside `reweight_entity()` (if needed) to confirm that the code path is being hit. Log `se == cfs_rq->curr` status, old and new weights, and whether `min_deadline_cb_propagate()` is called. This helps verify the driver is correctly triggering the buggy path.

### Important Note on the Parent-Level Tree

The heap corruption occurs in the **parent** CFS runqueue's rb-tree, not the group-internal one. The group scheduling entity (`tg->se[cpu]`) lives in the root (or parent) cfs_rq's rb-tree. When validating, check the parent cfs_rq's tree. For a single-level cgroup hierarchy, this is `cpu_rq(1)->cfs`. The `cfs_rq->tasks_timeline` at the root level contains the group scheduling entities.
