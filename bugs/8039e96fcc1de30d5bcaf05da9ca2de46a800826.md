# Fix forced idle sibling starvation corner case

- **Commit:** 8039e96fcc1de30d5bcaf05da9ca2de46a800826
- **Affected file(s):** kernel/sched/core.c, kernel/sched/fair.c, kernel/sched/sched.h
- **Subsystem:** core

## Bug Description

When core scheduling is enabled and a sibling CPU is forced idle (idle-selected to mitigate L1TF vulnerabilities), a single long-running task on the other sibling can starve the forced-idle sibling indefinitely. The forced-idle sibling does not get a chance to run because no scheduling event occurs to trigger re-evaluation of the core scheduling decision. This results in unfair CPU time distribution where one sibling is fully utilized while the other cannot execute any tasks.

## Root Cause

The scheduler relies on scheduling events (task wakeup, preemption, etc.) to re-evaluate the forced idle state and allow starved siblings to run. However, when a single task is actively running on one sibling with no interruptions, no such events occur. The scheduler lacks a mechanism to detect during periodic tick events whether a forced-idle sibling is being starved by the other sibling's long-running task.

## Fix Summary

The fix adds a `task_tick_core()` function that checks during each scheduler tick if a sibling is forced idle and the current runqueue has only one task that has already consumed its fair time slice. If these conditions are met, the running task is rescheduled to give the forced-idle sibling an opportunity to execute. Additionally, the `core_forceidle` flag is moved to shared state (core structure) rather than per-runqueue state to better reflect its scope in core scheduling.
