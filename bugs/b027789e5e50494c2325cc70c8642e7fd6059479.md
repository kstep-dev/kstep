# sched/fair: Prevent dead task groups from regaining cfs_rq's

- **Commit:** b027789e5e50494c2325cc70c8642e7fd6059479
- **Affected file(s):** kernel/sched/autogroup.c, kernel/sched/core.c, kernel/sched/fair.c, kernel/sched/rt.c, kernel/sched/sched.h
- **Subsystem:** CFS (task group scheduling)

## Bug Description

A use-after-free crash occurs in update_blocked_averages() when task groups are being destroyed. The bug manifests as cfs_rq structures remaining linked (on_list=1) to a task group that is in the process of being freed. This leads to accessing already-freed memory, causing kernel crashes with symptoms like general protection faults (#GP).

## Root Cause

The bug is a race condition between the task group cleanup path and the CFS bandwidth timer interrupt handler. When sched_offline_group() unlinks a task group, a timer interrupt via sched_cfs_period_timer() can trigger walk_tg_tree_from() which still sees the (not-yet-unlinked) dying task group. The timer handler's tg_unthrottle_up() then re-adds cfs_rq structures to this dying task group via list_add_leaf_cfs_rq(), while unregister_fair_sched_group() is concurrently trying to unlink them. These re-added cfs_rq's are then freed while still being linked, causing UAF in later updates.

## Fix Summary

The fix separates task group cleanup into two phases using RCU synchronization. First, sched_release_group() unlinks the task group from the global list early to prevent new RCU walkers from seeing it. Then, after an RCU grace period, sched_unregister_group() is called to safely unregister fair/RT bandwidth resources and free the cfs_rq/rt_rq structures. A second RCU grace period ensures that concurrent print_cfs_stats() calls always see valid objects.
